# PRD Review Skill

作为资深产品负责人，对 PRD / 产品需求文档 / 产品方案做结构化评审，输出评分 + 逐条修改建议（**不直接改文档**，只给修改建议；如果用户在报告后明确要求「直接改」，再用 Edit 落地）。

## 输入识别

按以下顺序确定要 review 的文件：

1. 用户消息里直接引用的文件路径或 `@filename`
2. IDE 当前打开 / 选中的文件（`ide_selection` 上下文）
3. 当前目录 / `docs/prds/` / `docs/` 下最近修改的 `.md` 文件（用 Glob 找，让用户确认）

如果用户在 skill args 或消息里提到「参考 XXX 规范」「按 YYY 模板」，把该规范文件也 Read 进来作为评分基线。常见规范位置：`docs/templates/`、`docs/specs/`、`docs/prd-template.md`、根目录 `PRD_TEMPLATE.md`。

## 评审维度与权重

总分 100 分，5 个维度：

| 维度 | 权重 | 核心问题 |
|------|------|---------|
| 1. 完整性 | 25 | 背景、目标、范围、用户故事、验收标准、风险、依赖是否齐全？ |
| 2. 清晰度与可测性 | 25 | 需求描述是否无歧义？验收标准是否可度量、可测试？ |
| 3. 用户价值与业务指标 | 20 | 痛点是否讲清？目标用户是否明确？成功指标是否可量化？ |
| 4. 边界条件与异常流 | 15 | 错误态、降级、并发、权限、空数据、兼容性是否覆盖？ |
| 5. 公司/产品规范符合度 | 15 | 是否遵循用户提供或仓库中的 PRD 模板、术语表、命名规范？若无明确规范，此项给基准分 12/15 并注明。 |

### 每个维度的检查清单

**1. 完整性（25 分）**
- [ ] 背景 / 问题陈述（4 分）
- [ ] 目标 & 非目标（4 分）
- [ ] 目标用户 & 典型场景（3 分）
- [ ] 功能范围（含 MVP 切分）（4 分）
- [ ] 用户故事 / 主流程（4 分）
- [ ] 验收标准（3 分）
- [ ] 依赖 & 风险（3 分）

**2. 清晰度与可测性（25 分）**
- [ ] 是否避免「支持」「完善」「优化」这类模糊词（5 分）
- [ ] 数量词是否明确（"快速" → "P95 < 200ms"）（5 分）
- [ ] 验收标准是 Given/When/Then 或等价可执行形式（8 分）
- [ ] 术语前后一致、没有同义词混用（4 分）
- [ ] 是否有示意图 / 表格 / 流程图（3 分）

**3. 用户价值与业务指标（20 分）**
- [ ] 是否讲清「不做会怎样」（5 分）
- [ ] 目标用户画像是否具体（4 分）
- [ ] 是否给出可量化成功指标（北极星 / 过程指标）（6 分）
- [ ] 是否定义了追踪口径（埋点、SQL、报表）（5 分）

**4. 边界条件与异常流（15 分）**
- [ ] 空态 / 初次进入 / 无数据（3 分）
- [ ] 错误态 / 超时 / 重试 / 降级（4 分）
- [ ] 权限 / 未登录 / 越权（3 分）
- [ ] 并发 / 幂等 / 竞态（3 分）
- [ ] 兼容性（旧版本、老数据、多端）（2 分）

**5. 公司/产品规范符合度（15 分）**
- [ ] 文档结构是否按模板（3 分）
- [ ] 术语表 / 名词统一（3 分）
- [ ] 版本号、作者、更新时间、变更记录（3 分）
- [ ] 关联文档交叉引用（架构图、API 契约、UI 稿）（3 分）
- [ ] 命名、大小写、中英混排规范（3 分）

等级：A 90-100 | B 75-89 | C 60-74 | D 40-59 | F <40

## 工作流

### Phase 1: 定位与上下文

1. 确认 review 目标文件（参见「输入识别」）
2. 若用户指定了参考规范，Read 规范文件
3. 扫一眼项目是否有 `docs/prds/` 目录或 `PRD` 模板文件（Glob `**/PRD*.md` 和 `docs/**/*.md` 前几条）；如有，把最近一份 PRD 作为风格参考（只参考，不照搬）
4. Read 目标文件全文

### Phase 2: 逐维度评分

按上面 5 个维度打分。**每个子项都必须在原文里找证据**（引用行号或原文片段）。没有证据 = 扣分。

### Phase 3: 输出评审报告

**一律使用下面的中文模板**，不要自由发挥：

```markdown
# PRD Review 报告

**文件**：`<相对路径>`
**总分**：XX / 100（等级：X）
**一句话总评**：……

## 维度评分

| # | 维度 | 得分 | 等级 |
|---|------|------|------|
| 1 | 完整性 | X / 25 | X |
| 2 | 清晰度与可测性 | X / 25 | X |
| 3 | 用户价值与业务指标 | X / 20 | X |
| 4 | 边界条件与异常流 | X / 15 | X |
| 5 | 公司/产品规范符合度 | X / 15 | X |

## 详细扣分项

### 1. 完整性 (X/25)
- ❌ **缺少「非目标」章节** — 未明确哪些不做，易引发范围蔓延
- ⚠️ **风险章节过简** ([docs/prds/xxx.md:42](docs/prds/xxx.md#L42)) — 只列了技术风险，未覆盖运营/合规
- ✅ 背景、目标、范围已完备

（2、3、4、5 同格式）

## 修改建议（按优先级）

### 🔴 P0（必须改，否则无法评审通过）
1. **[完整性]** 补「非目标」章节，明确本期不做的事项
   - 位置：建议放在「目标」之后、「范围」之前
   - 参考写法：`> 非目标：本期不支持 XX；不处理 YY 场景（下一期）`
2. **[清晰度]** [xxx.md:L88](xxx.md#L88) 「系统应快速响应」改为具体 SLA
   - 建议：`API 响应 P95 < 200ms，P99 < 500ms`

### 🟡 P1（强烈建议改）
...

### 🟢 P2（锦上添花）
...

## 亮点
- ……

## 下一步
本报告只给出建议。如需我直接落笔改到文档，请回复「直接改」并指明要采纳哪些建议（默认 P0 全采纳）。
```

### Phase 4（可选）：落笔修改

只有用户明确说「直接改」「帮我改」「按建议改」之后才进入。按以下规则：

- 用 Edit 工具，一次一个 old_string/new_string
- P0 默认全改；P1/P2 等用户勾选
- 每个修改前简述「要改什么、为什么」
- 保留原文档风格（标题级别、编号、emoji 用法）
- 不要删用户的原文，除非明显错误或用户要求

## 额外规则

- **引用必须带行号**：`[文件名:行号](相对路径#Lxx)`，方便 PM 跳转
- **不要假想证据**：没读到就不打分，别脑补
- **不要粘大段原文**：引用时只截关键片段（不超过 2 行）
- **对于英文 PRD**：评审报告仍用中文输出，但修改建议里的术语保持原文
- **多文件 PRD**：如果 PRD 是一个目录（含 `README.md` + 子章节），分别打分，末尾再给整体分（加权平均）
- **避免套话**：「建议优化」「有待完善」是废话，必须给出具体改写方案
