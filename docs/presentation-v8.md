## P1 · 需求痛点（按研发全链路梳理）

| 阶段 | 核心痛点 |
|------|---------|
| **设计** | PRD 质量参差、评审靠人肉通读；规范散落在文档 / 老员工脑中，新人 onboarding 难 |
| **开发** | 代码习惯不一，多语言 / 多租户 / 安全规范执行不一致；AI 每个场景都要重写 prompt，能力无复用 |
| **评审** | MR review 耗时主观，质量标准无法机器化校验 |
| **测试** | 覆盖率无硬约束、测试基础设施参差（脚本缺失、镜像不统一），发布后再救火 |
| **运维** | 部署 / 回滚 / 日志 / MR 散落在 Jenkins、GitLab、K8s、Harbor，上下文切换频繁 |
| **全链路** | **AI 不懂业务**（术语用错、合规漏看、bug 和业务预期分不清）；**经验难沉淀**（同类错误反复犯） |

**一句话总结**：研发链路既缺**硬门禁**，也缺**软知识**，更缺**自我进化**能力。

## P2 · 设计理念

### 核心理念一句话

> **让 AI 做能做的，让人做该做的，规范和测试做裁判，经验让平台自我进化，Skills 让能力可组合复用，行业知识让 AI 真的懂行。**

### 三大质量承诺

| 承诺 | 含义 |
|------|------|
| **不过门禁，到不了生产** | 规范 + 测试 + 评审三道卡点，自动拒绝不合格代码 |
| **同一错误，不犯第二次** | 所有失败经验沉淀到学习中枢，反哺下一次 AI 行为 |
| **能力复用，不造轮子** | Skills 商店 + 行业知识库让已有能力跨产线、跨项目复用 |

### 方法论四句话

- **规范显性化** — 所有规范以 markdown 存 GitLab 知识库，AI 随时读取、改 MD 即生效
- **文档入 Git** — PRD / 设计文档作为代码库一等公民，走 MR 卡点
- **测试强约束** — 覆盖率 + 脚本 + 镜像纳入公司规范，三道门禁（编译 / MR / 发布）
- **自我迭代** — 事件 → 入库 → 汇总 → 产物（prompt 补丁 / 回归用例 / 反模式）→ 注入下一次 AI

## P4 · 管控方案 · 五大中枢

| 中枢 | 定位 | 内容 | 使用方式 |
|------|------|------|----------|
| **规范中枢** | 硬约束 · 写什么 | 公司级（多语言 / 多租户 / 认证 / UX / UI / 日志 / 测试 / 脚本规范）+ 产品级（领域模型 / 术语 / API 契约） | GitLab markdown，审核时动态加载，改 MD 即生效 |
| **测试中枢** | 硬门禁 · 够不够 | 覆盖率硬约束（总体 ≥70% / 核心 ≥90%）+ 脚本规范（`scripts/{build,package,test}.sh` + Dockerfile 模板） | 三道门：编译门（build.sh 内嵌）/ MR 门 / 发布门 |
| **学习中枢** | 经验沉淀 · 怎么更好 | 回归用例库 · 项目 prompt 补丁 · 反模式手册 · 规范自生长 · Agent 盲区 · override 样本 | 事件驱动入库，周期性汇总，注入下一次 AI 上下文 |
| **Skills 商店** | 能力复用 · 怎么做漂亮 | 6 大分类（设计 / 开发 / Bug / 测试 / 评审 / 运维）· 可组合能力包（prompt + 工具模式 + 示例） | 4 层绑定（全局 / 产线 / capability / 场景）· GitLab 仓库导入 + 版本管理 |
| **行业知识库** | 业务懂行 · 这是什么业务 | 4 分类（术语词典 / 法规合规 / 业务流程 / 最佳实践）· 跨产线共享 | 产线 ↔ 行业多对多绑定 · 简单全量注入 / 进阶 RAG 检索 |

**AI 上下文组装顺序**（五中枢如何协同注入）：

```
1. 基础 system prompt
2. 公司规范 MD        ← 规范中枢
3. 产品规范 MD        ← 规范中枢
4. 行业知识库片段     ← 行业知识库
5. Skills 组合        ← Skills 商店
6. 项目 prompt 补丁   ← 学习中枢
```

## P5 · 管控方案 · MR 五维质量门

**核心抓手**：代码进主干前，Pipeline 自动并发五维审核，不过则自动修复重审。

```
┌───────────────────────────────────────────────┐
│              MR 五维质量门                     │
├───────────────────────────────────────────────┤
│ ① 需求匹配     代码 ↔ Story 验收标准          │
│ ② 代码质量     可读性 / 复杂度 / 命名          │
│ ③ 安全         注入 / 泄露 / 依赖漏洞 / 认证   │
│ ④ 规范合规     公司规范 + 行业合规（动态加载）  │
│ ⑤ 自动化测试   覆盖率 + 测试通过 + ATDD 执行   │
└───────────────────────────────────────────────┘
          ↓
     全部通过？
     /        \
    是          否
    ↓           ↓
  Merge     自动修复 Agent
              ↓
         改代码 / 补测试 / 提 Bug Issue
              ↓
           push → 重审 ↺
              ↓
         3 次仍不过 → 群通知人工接管
```

**测试失败专用闭环**：测试失败 → Bug Agent 开合规 Issue → Fix Agent 修复 → push → 重审，**Bug 修复 MR 必须带 E2E 回归用例**。

**发布前质量门**：全量部署测试 + E2E + 回归测试 → 全过才允许发版 → 失败自动回滚。

## P6 · MVP 功能清单（能跑起闭环）

**目标**：**让一条产品线，从 PRD 到 MR 合并，规范 + 测试两道门跑通**

> 标记说明：**✅ = 已实现** · **⬜ = 待实现**

### 核心功能

**设计阶段**
- ✅ PRD Agent + 架构设计 Agent 多轮对话生成
- ⬜ PRD / 设计文档入项目 Git（约定 `docs/prd/*.md`、`docs/arch/*.md`）
- ⬜ 文档 MR 卡点 + AI 评审

**规范中枢（MVP 档）**
- ⬜ 公司级规范 markdown 托管 GitLab 知识库 + 动态加载
- ⬜ 基础工具：`load_company_standards` / `check_standard_compliance`

**开发阶段**
- ✅ Claude Code 本地手动开发（`create_mr` / `submit_review` / `read_code` 已就绪）

**MR 质量门（4 维 MVP 档：需求 / 代码 / 安全 / 测试）**
- ✅ 原子工具 `review_mr_diff`（单点 AI 评审）
- ⬜ Pipeline 化并发 4 维审核（编排 `mr-quality-gate` 模板）
- ⬜ 公司规范维度用**静态规则**（V2 接动态加载）
- ⬜ 自动修复闭环（原子工具 `fix-code` ✅，"MR 失败 → 自修 → 重审" 串联 ⬜）

**测试中枢**
- ✅ 原子工具 `autotest` + `run_tests`
- ⬜ 公司级测试基础设施规范（scripts 四件套 + Dockerfile + 覆盖率）
- ⬜ 覆盖率硬约束：总体 ≥ 70% · 核心 ≥ 90%
- ⬜ **编译即测试**（build.sh 内嵌，失败 = 编译失败）

**测试失败自愈**
- ✅ Bug 分析 Agent L1-L3 自动修复（fix-runner + MR 对账）
- ✅ Pipeline 失败分析 `failure-analyzer`
- ⬜ 测试失败 → Bug Issue → Fix → 重审 串联

**自迭代（最高 ROI 三项）**
- ⬜ **A1** Bug 修复强制附 E2E 回归用例
- ⬜ **B5** Claude 异常 → 模型降级（Opus → Sonnet → Haiku）
- ⬜ **D1** 人工 override 记录 → 样本库（为 V2 学习积累数据）

**平台基建（已备齐）**
- ✅ IM 双平台 · 审批工作流 · Pipeline 引擎 · IM 驱动流水线 · 管理后台 · 26 版数据库 · Docker 两层 + GitLab CI

### MVP 成功标准

- 试点产品线跑通完整闭环（PRD → MR → 测试 → 合并）
- MR 质量门自动审核率 ≥ 70%
- 测试失败自愈成功率 ≥ 50%
- 覆盖率不达标 MR **100% 拦截**
- MR 平均评审时长下降 ≥ 30%

**MVP 不做**：平台托管开发 · 发布前门禁 · Skills 商店 · 行业知识库 · 多 IM 扩展

## P7 · V2 功能清单（闭环增强 + 学习启动）

**目标**：**开发自动化进场 + 发布强门禁 + 平台从"执行"开始"学习"**

**平台托管开发 MVP**

- ATDD 用例生成（Given/When/Then，含非功能验收）

**规范中枢完整版**
- 产品级规范加入（per 产品线领域模型 / 术语 / API 契约）
- 规范版本锚定 commit sha

**MR 质量门完整 5 维**
- 第 ④维公司规范接入动态加载
- 规范违反 → 自动修复 → 重审闭环打通

**发布前质量门**
- 全量部署测试 Pipeline 模板（`release-qa-gate`）
- E2E 测试编排 + 回归测试库
- 发布负责人 + QA 会签
- 失败 → 自动回滚 + Bug Issue

**学习中枢激活（5 项自迭代）**
- **A2** PRD commit 历史 → 项目 prompt 补丁
- **A3** MR 评审失败聚类 → 代码反模式手册
- **A4** 规范违反 → 规范库自生长（AI 向知识库提 MR）
- **B3** 部署失败 → 自动回滚 + 根因 Bug
- **D4** 探索测试反馈 → E2E 用例孵化

**Skills 商店 MVP**
- 6 大分类预置 Skills（约 20+ 条）
- 产品线级绑定
- 基础动态注入链路
- 管理后台浏览 + 绑定配置页

**学习中枢看板**
- 回归用例增长曲线 · prompt 补丁版本 · override 热力图 · 发布质量评分卡

### V2 成功标准

- 3+ 产品线跑通完整 5 维质量门
- 平台托管开发在 1 条产线完成 MVP 验证
- 发布质量评分卡覆盖所有发版
- Skills 预置 20+，产线平均绑定 ≥ 5 条
- 学习中枢产出：项目 prompt 补丁月均 ≥ 10 条、规范 MR 月均 ≥ 3 条

## P10 · 成功度量与工程规范

### 成功度量（跨阶段共用）

- **效率**：MR 平均评审时长 · PRD → 合并 lead time · 自动修复成功率
- **质量**：生产事故率 / 回滚率 · 覆盖率达标产线数 · 规范违反拦截率
- **使用**：月活跃产线 / 用户 · 日均 IM 触发 · Skills / 规范 / 行业库引用次数
- **学习**（V2 起）：项目 prompt 补丁月产出 · 规范 MR 月产出 · 反模式条目增长
- **成本**（V3）：Claude Token / 产线 / MR · 任务耗时 · ROI 评估

### 产品项目必须遵守的公司级规范

| 类别 | 强制要求 |
|------|----------|
| 文档 | PRD / 设计文档 markdown 进项目 git，走 MR 评审 |
| 规范载体 | 公司 / 产品规范 markdown 存 GitLab 知识库 |
| 测试入口 | `./scripts/test.sh` 统一入口，不分语言 |
| 测试覆盖率 | 总体 ≥ 70%，核心 ≥ 90% |
| 编译脚本 | **`./scripts/build.sh` 必须跑测试，测试失败 = 编译失败** |
| 基础镜像 | `Dockerfile.base` / `Dockerfile.test` 必备 |
| Bug 修复 | **修复 MR 必须带 E2E 回归用例** |
| Skill 导入 | 外部 Skill 必须走 GitLab 或签名包（V2+） |
| 行业知识 | 行业库必须 GitLab 托管 + 敏感库 RBAC 限定（V3+） |

### 关键风险与前置条件

- **MVP 前置**：试点产品线选定 · 公司规范文档团队配合 · 覆盖率工具到位
- **V2 前置**：MVP 运行 3+ 月积累数据 · 托管开发团队培训
- **V3 前置**：V2 稳定节奏 · 行业知识整理 owner 到位

## P11 · 总结

### 三阶段的价值放大点

- **MVP 解决"能用"**：标准化 —— 把研发做成有规律的动作
- **V2 解决"好用"**：自动化 —— 让机器做更多，让人做更聪明的事
- **V3 解决"行业用"**：生态化 —— 平台变成知识与能力的沉淀载体
