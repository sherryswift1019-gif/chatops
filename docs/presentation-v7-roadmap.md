## P1 · 阶段划分的三个判断原则

**为什么这样切分？**

1. **MVP = 能跑起闭环**
   - 必须覆盖"从 PRD 到 MR 合并"的主路径
   - 每个环节有**硬卡点**（不过不能进下一步）
   - 复用现有能力为主，新增能力克制
   - 交付给内部产品线试点可用

2. **V2 = 闭环变强 + 学习启动**
   - MVP 跑一段后**数据积累**起来，学习中枢才有输入
   - 开发侧引入平台托管模式，研发效率二次提升
   - 发布侧补强门禁，从"能交付" → "敢发布"

3. **V3 = 生态化 + 行业懂行**
   - Skills 商店完整版让能力**可复用可分享**
   - 行业知识库让平台**跨行业适配**
   - 高级自迭代能力需要长期数据支撑

## P3 · MVP：能跑起闭环

**目标**：一句话——**让一条产品线，从 PRD 到 MR 合并，规范 + 测试两道门**

### 必选能力清单

> 标记说明：**✅ = 已实现** · **⬜ = 待实现**（MVP 要补齐的内容）

**设计阶段**
- ✅ PRD Agent + 架构设计 Agent 多轮对话生成（`prd_documents` / `arch_documents` 及配套工具已就位）
- ⬜ PRD / 设计文档入项目 Git（目前 PRD 落在 DB，需约定 `docs/prd/*.md`、`docs/arch/*.md` 并同步到项目仓库）
- ⬜ 文档 MR 卡点 + AI 评审（需在 `review_mr_diff` 基础上扩展"文档类 MR"的评审 Agent）

**规范中枢（MVP 档）**
- ⬜ 公司级规范 markdown 托管 GitLab 知识库（`product_knowledge_repos` 是仓库索引，缺规范动态加载机制）
- ⬜ 审核触发时动态加载（改 MD 即生效）
- ⬜ 基础工具：`load_company_standards`、`check_standard_compliance`

**开发阶段**
- ✅ Claude Code 本地手动开发（`create_mr` / `submit_review` / `read_code` 等工具已就绪，支持完整 MR 流程）
- ⬜ 平台托管开发（Epic/Story/ATDD）→ **V2 引入**

**MR 质量门（4 维 MVP 档）**
- ✅ 原子工具 `review_mr_diff`（单点 AI 评审能力已有）
- ⬜ Pipeline 化并发 4 维审核（①需求 ②代码 ③安全 ⑤测试，需编排成 `mr-quality-gate` 模板）
- ⬜ ④公司规范维度用**静态规则**（MVP 档，延后到 V2 接动态加载）
- ⬜ 不过 → 自动修复闭环（原子工具 `fix-code` ✅，但 "MR 失败 → 自动修复 → 重审" 闭环未打通）
- ⬜ 达上限 3 次 → 群通知人工接管

**测试中枢**
- ✅ 原子工具 `autotest`（测试流水线触发） + `run_tests`（命令级测试执行）
- ⬜ 公司级测试基础设施规范（`scripts/{build,package,test}.sh` + Dockerfile + 覆盖率约束）
- ⬜ 覆盖率硬约束：总体 ≥ 70% · 核心 ≥ 90%
- ⬜ **编译即测试**（build.sh 内嵌，失败 = 编译失败）
- ⬜ MR 门自动跑测试 + 覆盖率
- ⬜ 发布前质量门（全量部署 + E2E）→ **V2 引入**

**测试失败自愈**
- ✅ Bug 分析 Agent L1-L3 自动修复（Bug 场景闭环已打通，含 `fix-runner` 串行执行 + MR 对账）
- ✅ Pipeline 失败分析 `failure-analyzer`（sonnet 驱动的根因建议已有）
- ⬜ **测试失败 → 合规 Bug Issue → Fix Agent → 重审** 闭环（原子能力齐备，缺串联）

**自迭代（最高 ROI 三项）**
- ⬜ **A1** Bug 修复强制附 E2E 回归用例
- ⬜ **B5** Claude 调用异常 → 模型降级（Opus / Sonnet / Haiku）
- ⬜ **D1** 人工 override 记录 → 样本库（为 V2 学习提供数据）

**平台基建**
- ✅ IM 双平台（钉钉 Stream / 飞书 Webhook）
- ✅ 审批工作流 Gate/Router/Escalation
- ✅ Pipeline 引擎（5 种 stage）+ IM 驱动流水线（schema-v19）
- ✅ Bug 分析 L1-L3 全自动（内置 Pipeline 模板 schema-v25）
- ✅ 管理后台基础页面（19 个 React 页面 + 27 个 Admin API）
- ✅ 26 版本数据库演进 + Docker 两层构建 + GitLab CI

### MVP 成功标准

- 试点产品线跑通一次完整闭环（PRD → MR → 测试 → 合并）
- MR 质量门自动审核率 ≥ 70%
- 测试失败自愈成功率 ≥ 50%
- 覆盖率不达标的 MR **100%** 被拦截
- MR 平均评审时长下降 ≥ 30%

### MVP 不做什么（明确排除）

- ❌ 平台托管开发（Epic/Story/ATDD）
- ❌ 发布前质量门
- ❌ Skills 商店
- ❌ 行业知识库
- ❌ 规范自生长 / 反模式手册 / 盲区手册（需数据积累）
- ❌ 多 IM 平台扩展

## P4 · V2：闭环增强 + 学习启动

**目标**：一句话——**开发自动化进场，发布门禁补齐，平台从"执行"开始"学习"**

### 新增能力清单

**开发阶段：平台托管模式 MVP**
- 🆕 Epic 拆分 + Epic Review Agent
- 🆕 Story 拆分 + Story Review Agent
- 🆕 ATDD 用例生成 Agent（Given/When/Then）
- 🆕 Story 实现（Claude Code 按 ATDD 驱动）+ Code Self-Review
- 🆕 托管 / 手动双模式可视化切换

**规范中枢完整版**
- 🆕 **产品级规范** 加入（per 产品线领域模型 / 术语 / API 契约）
- 🆕 规范版本锚定 commit sha（MR 可追溯"规范 @ sha"）

**MR 质量门完整 5 维**
- 🆕 ④公司规范维度接入规范中枢动态加载
- 🆕 规范违反 → 自动修复 → 重审闭环打通

**发布前质量门**
- 🆕 全量部署测试 Pipeline 模板
- 🆕 E2E 测试编排 + 回归测试库
- 🆕 发布负责人 + QA 会签
- 🆕 失败 → 自动回滚 + Bug Issue
- 🆕 QA 角色转型：探索性测试 + 真实场景 E2E 设计

**自迭代（V2 激活 5 项）**
- 🆕 **A2** PRD commit 历史 → 项目 prompt 补丁
- 🆕 **A3** MR 评审失败聚类 → 代码反模式手册
- 🆕 **A4** 规范违反 → 规范库自生长（AI 向知识库提 MR）
- 🆕 **B3** 部署失败 → 自动回滚 + 根因 Bug
- 🆕 **D4** 探索测试反馈 → E2E 用例孵化

**Skills 商店 MVP**
- 🆕 6 大分类预置 Skills（约 20+ 条）
- 🆕 产品线级绑定（capability / 场景级延后到 V3）
- 🆕 基础动态注入链路（基础 prompt + 规范 + Skills + 学习补丁）
- 🆕 管理后台 Skills 浏览 + 绑定配置页

**学习中枢看板**
- 🆕 回归用例增长曲线
- 🆕 项目 prompt 补丁版本历史
- 🆕 override 热力图
- 🆕 发布质量评分卡

### V2 成功标准

- **3+ 产品线**跑通完整 5 维质量门
- 平台托管开发在 **1 条产品线** 完成 MVP 验证
- 发布质量评分卡覆盖所有发版
- Skills 商店预置 **20+ Skills**，产线平均绑定 **≥ 5 条**
- 学习中枢产出：项目 prompt 补丁月均 **≥ 10 条** · 规范 MR 月均 **≥ 3 条**
- 发布前门禁拦截率数据有初步基线

### V2 依赖 MVP 的什么

- **D1 样本库**（MVP 积累）→ V2 学习中枢训练数据
- **A1 回归库**（MVP 积累）→ V2 发布前门禁的回归测试来源
- **MR 质量门 4 维**（MVP 跑通）→ V2 扩展第 5 维

## P5 · V3：生态化 + 行业懂行

**目标**：一句话——**能力可组合可分享，平台跨行业适配，从工具变成研发生态**

### 新增能力清单

**Skills 商店完整形态**
- 🆕 4 层绑定完整（全局 → 产品线 → capability → 场景）
- 🆕 GitLab 仓库导入 + 打包导入 + 版本管理
- 🆕 Skills 编排 Agent（根据任务类型自动挑 skills）
- 🆕 使用度量看板（触发次数 / 成功率 / 耗时）
- 🆕 Bug L1-L4 各自绑定专属 Skills

**行业知识库（v6 全量）**
- 🆕 4 分类全覆盖：术语词典 / 法规合规 / 业务流程 / 最佳实践
- 🆕 GitLab 仓库托管 + PDF/Word 提取 + 外链快照
- 🆕 产品线 ↔ 行业多对多绑定
- 🆕 动态注入（上下文第 4 层）
- 🆕 **RAG 检索**（大型法规 / 白皮书语义召回）
- 🆕 行业合规审核 Agent（MR 质量门行业子项）
- 🆕 引用溯源审计

**高级自迭代**
- 🆕 **A5** Agent 盲区手册（连续失败 N 次登记）
- 🆕 **C1** 审批规则通过率监控 → 建议降级
- 🆕 **C3** 工具调用热度 → 死工具建议下线

**多平台与隔离**
- 🆕 企业微信 / Slack 适配器
- 🆕 多租户数据隔离
- 🆕 跨产线权限沙箱

**可观测与成本**
- 🆕 Claude Token 成本看板（per 产线 / per 任务类型）
- 🆕 执行耗时分布 + 失败率监控
- 🆕 Skills / 规范 / 行业库的 ROI 度量

**生态化（愿景）**
- 🎯 Skill 社区 / 跨团队分享
- 🎯 行业知识库跨公司共享（开放生态）
- 🎯 第三方 Skill / 行业库市场

### V3 成功标准

- Skills 商店覆盖 **N+ 个 Skills** · 月新增 **M+**
- 行业知识库覆盖 **3+ 行业** · 引用率 **≥ Z%**
- 行业合规审核拦截率有稳定基线
- 跨产线 Skill 复用率 **≥ W%**
- 成本看板支持月度对比 + 趋势预警
- 多 IM 平台至少上线 1 个额外平台

### V3 依赖 V2 的什么

- **A3 反模式手册**（V2 积累）→ V3 孵化 Skill 的原料
- **Skills 商店 MVP**（V2）→ V3 完整形态的基础
- **学习中枢数据**（V2）→ V3 盲区手册与度量驱动

## P7 · 风险、假设与前置条件

### 关键假设

| 假设 | 风险等级 | 验证方式 |
|------|---------|---------|
| 团队愿意把 PRD 进项目 git | 中 | MVP 试点 1 条产线先验证 |
| 公司规范可以 markdown 化 | 低 | 文档团队配合，先做 TOP 10 规范 |
| 覆盖率 70% / 90% 可达 | 中 | 试点产线先摸底，必要时调整阈值 |
| Claude Opus 配额足够 | 中 | 监控 Token 消耗，V3 做成本看板 |
| 托管开发 ATDD 模式可收敛 | 高 | V2 先小范围试点 1 个功能 |

### 前置条件

- **MVP 前置**：试点产品线确定、规范文档团队到位、覆盖率工具到位
- **V2 前置**：MVP 运行 3+ 个月积累数据、托管开发团队培训
- **V3 前置**：V2 形成稳定节奏、行业知识整理 owner 到位

### 需要对齐的决策

- MVP 试点产品线选哪个？
- 公司规范的维护责任归谁？
- Skill / 行业库的审核上线流程？
- 自动修复失败的人工接管值班机制？

## P8 · 成功度量（每阶段共用口径）

**效率指标**
- MR 平均评审时长（下降率）
- PRD → 合并的 lead time
- 自动修复成功率

**质量指标**
- 生产事故率 / 回滚率
- 覆盖率达标产线数
- 规范违反拦截率

**使用指标**
- 月活跃产线数 / 月活跃用户数
- 日均 IM 触发次数
- Skills / 规范 / 行业库引用次数（V2-V3）

**学习指标（V2 起）**
- 项目 prompt 补丁月产出
- 规范 MR 月产出
- 反模式手册条目增长

**成本指标（V3）**
- Claude Token 成本 / 产线 / MR
- 平均任务耗时
- ROI 评估

## P11 · 总结

**MVP 解决"能用"**：规范 + 测试驱动的 MR 合并闭环，跑通最小价值链路
**V2 解决"好用"**：开发自动化进场 + 发布强门禁 + 平台开始自学习
**V3 解决"行业用"**：能力可组合可分享 + 行业懂行 + 生态化可延展

每一阶段的**价值放大点**：
- MVP 是**标准化**：把研发做成有规律的动作
- V2 是**自动化**：让机器做更多、让人做更聪明的事
- V3 是**生态化**：平台变成知识与能力的沉淀载体

> 🎯 三阶段之后的终极形态：**一个能跑遍研发全流程、懂公司规范、懂行业业务、会自己学习、能跨团队共享能力的研发生态平台。**
