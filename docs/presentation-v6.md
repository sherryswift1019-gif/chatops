## P1 · 项目定位（v6 升级）

**一句话**：覆盖 **PRD 设计 → 代码开发 → MR 审核 → 测试 → 部署运维** 的全流程平台，让 AI **懂代码、懂规范、懂业务、会学习**。

- **四大阶段 + 五大中枢**：
  1. **设计 / 开发 / 评审 / 运维** 四阶段
  2. **规范中枢** — 公司 / 产品 markdown 规范（硬约束）
  3. **测试中枢** — 覆盖率 / 脚本规范（三道门禁）
  4. **学习中枢** — 历史经验沉淀（自我迭代）
  5. **Skills 商店** — AI 能力包动态装配
  6. **行业知识库**（v6 新）— 业务领域上下文，让 AI 懂行
- **质量承诺**：不过门禁到不了生产、同一错误不犯第二次、能力复用不造轮子、**懂行业不说外行话**

## P2 · 解决什么问题

**研发全链路痛点**

- PRD 质量参差、规范散落、代码习惯不一
- 测试覆盖率无硬约束、基础设施参差
- MR review 耗时主观，标准无法机器化
- 经验难沉淀，同类错误反复犯
- AI 能力碎片化，每个场景重写 prompt
- **AI 不懂业务**：写 PRD 用错术语、审代码忽略行业合规、分析 bug 分不清"业务预期"和"真 bug"（v6 新增痛点）
- 运维散落多系统，上下文切换频繁

**Mewtwo 的切入点**

- 规范显性化 + 测试强约束 + 文档入 Git + 双通道开发 + 五维 MR 门禁 + 发布前质量门
- 自我迭代：9 项知识沉淀 + 运行时自愈 + 协作反馈
- Skills 可组合：6 大类能力包按场景动态装配
- **懂行业**（v6 新）：行业知识库注入业务术语 / 法规 / 领域模型 / 最佳实践，AI 用业务语言说话
- 运维对话化：IM 触发流水线 + 审批超时升级

## P6 · 规范审核中枢

**核心理念**：规范不是文档，是**可执行的校验器**。

- **公司级**：多语言 / 多租户 / 认证防护 / UX / UI / 日志 / 错误码 / API 契约 / 测试规范 / 脚本规范
- **产品级**：领域模型 / 术语词典 / 权限模型 / 命名约定
- **加载**【规划】：GitLab markdown → 动态注入 → 改 MD 即生效、commit sha 锚定
- **4 大卡点**：PRD 评审 / Epic-Story 拆分 / MR 代码 / MR 测试

## P7 · PRD & 设计评审闭环

**原则**：不管用户怎么写，只审核最终 markdown

```
编写 → 文档进 Git → MR → AI 评审（完整性 / 规范 / 可实现性 / 行业合规）
     → 人工会签 → 不通过改 → 重审 → 合入 → 作为开发输入
```

## P16 · 行业知识库（v6 新增 · 核心模块）

**核心理念**：让 AI **懂业务**。规范告诉 AI 怎么写，Skills 告诉 AI 怎么做，**行业知识库告诉 AI 这是什么业务**——术语、法规、流程、最佳实践，都是 AI 的业务背景。

### 与现有知识体系的边界

| 层级 | 作用域 | 典型内容 | 存储 |
|------|--------|----------|------|
| **公司规范** | 全公司 | 代码 / 文档 / 脚本规范 | GitLab 知识库 |
| **产品规范** | per 产品线 | 领域模型 / API 契约 / 命名 | GitLab 知识库 |
| **产品私有知识** | per 产品线 | 这个产品怎么运作 | `product_knowledge_repos`【现有】 |
| **行业知识库**（v6） | per 行业（**多产线共享**） | 行业术语 / 法规 / 流程 / 白皮书 | GitLab 仓库 + 可选向量库 |

**关键差异**：产品知识是**产品自己的事**（私有）；行业知识是**整个行业的事**（共享）。例如"PAM 产品的权限模型"是产品知识，"电力监控行业调度规程"是行业知识。

### 4 个核心分类维度

| 维度 | 示例 | 使用场景 |
|------|------|----------|
| **术语词典** | 行业缩写 / 专业名词 / 中英对照 | PRD 评审术语一致性、代码命名审核 |
| **法规合规** | 数据安全 / 隐私保护 / 行业标准（GB/ISO） | MR 合规维度审核（P10 ④扩展） |
| **业务流程** | 核心业务的标准流转 / 关键节点 / 角色 | PRD / 架构评审的业务合理性 |
| **最佳实践** | 行业成熟方案 / 参考架构 / 反模式 | 架构设计、Bug 根因分析 |

### 来源与导入

- **GitLab 行业仓库**（推荐）：以仓库形式托管，markdown 为主
- **文件上传**【规划】：PDF / Word / Excel → 自动提取为 markdown
- **外链索引**：法规官网 / 行业协会 URL，定期抓取快照
- **版本管理**：每次更新锚 commit sha，支持版本回滚

### 绑定机制

```
行业知识库  ⇌  产品线（多对多）
   │
   └ PAM 产品线 → [电力监控行业, 工业控制安全]
   └ XYZ 产品线 → [金融合规, 数据隐私]
```

**多行业叠加**：一个产品可以同时绑定多个行业库（如金融 + 数据隐私），加载时合并。

### 检索与注入

**两档方案**【规划】

- **简单档**：按绑定全量注入关键术语表 + 合规要点（适合体量小的行业库）
- **进阶档 RAG**：长文 + 非结构化内容向量化，按语义召回相关片段（适合大型法规 / 白皮书）

**注入上下文顺序（v6 更新）**

```
1. 基础 system prompt
2. 公司规范 MD
3. 产品规范 MD
4. 【行业知识库 v6】术语 + 合规要点 + RAG 召回
5. Skills 组合（v5）
6. 项目 prompt 补丁（v4 学习中枢）
```

### 使用场景（覆盖全流程）

- **PRD 评审**：检查业务术语、流程合理性、法规遗漏
- **架构设计**：参考行业成熟方案、识别反模式
- **MR 代码评审**：行业合规维度（数据处理、审计日志、加密要求）
- **Bug 分析**：区分"业务预期"vs"真 bug"，引入行业上下文
- **Skill 增强**：行业类 Skill（如"电力调度规程"）基于知识库派生

### 权限与审计

- **行业库级 RBAC**：部分商业敏感库限定产品线可见
- **访问审计**：每次注入记录到 `audit_log`
- **签名导入**：外部库导入需走 GitLab 或签名包（防供应链攻击）

### 管理后台【规划】

- **行业库首页**：按行业分类浏览 + 搜索
- **库详情**：版本 / 来源 / 目录 / 关联产线
- **绑定配置**：产品线 ↔ 行业多对多可视化
- **使用度量**：每次 AI 任务引用了哪些行业片段（可追溯）

### 与五大中枢的关系

| 中枢 | 与行业知识库的协同 |
|------|-----|
| 规范中枢 | 规范可引用行业法规条款，保持与行业同步 |
| 测试中枢 | 行业级合规测试用例沉淀进发布回归库 |
| 学习中枢 | 行业知识更新 → 触发相关 prompt 补丁重建 |
| Skills 商店 | 行业类 Skill 依赖行业知识库作为输入 |

- `resolveGitlabConfig()` 统一入口、Tool 自注册、迁移幂等、命名对齐、前端表单 Select 化

**产品项目必须遵守的公司级规范**

| 类别 | 强制要求 |
|------|----------|
| 文档 | PRD / 设计文档 markdown 进项目 git，走 MR 评审 |
| 规范载体 | 公司 / 产品规范 markdown 存 GitLab 知识库 |
| 测试入口 | `./scripts/test.sh` 统一入口 |
| 测试覆盖率 | 总体 ≥ 70%，核心 ≥ 90% |
| 编译脚本 | **`./scripts/build.sh` 必须跑测试，测试失败 = 编译失败** |
| 打包 / 镜像 | `package.sh` + `Dockerfile.{base,test}` + `test-setup.sh` |
| Bug 修复（v4） | **修复 MR 必须带 E2E 回归用例** |
| Skill 导入（v5） | 外部 Skill 必须走 GitLab 或签名包 |
| **行业知识（v6）** | **行业库必须 GitLab 托管 + 敏感库 RBAC 限定** |

## P28 · 价值、成果与规划

**已实现价值**

- 统一运维入口（IM）· AI 辅助设计（PRD / 架构）· Bug 自动化（L1-L3）· 企业级兜底

**v2 规划**：规范中枢 · PRD/设计 MR 卡点 · 托管开发 MVP · MR 五维质量门

**v3 规划**：测试基础设施规范化 · MR 测试门禁 · 发布前质量门 · 编译即测试

**v4 规划**：9 项自迭代（Bug→E2E / PRD→prompt / 反模式手册 / 规范自生长 / 盲区 / 部署自愈 / 模型降级 / override / E2E 孵化）

**v5 规划**：Skills 商店（6 大类 + 4 层绑定 + 动态注入 + GitLab 导入 + 度量）

**v6 规划**（本次新增）

- **行业知识库 MVP** — GitLab 仓库托管 + 4 大分类（术语 / 法规 / 流程 / 最佳实践）
- **产品线 ↔ 行业绑定** — 多对多配置
- **动态注入** — 上下文组装加入行业知识片段
- **RAG 检索**【愿景】— 大型法规 / 白皮书按语义召回
- **行业合规审核** — MR 五维中的公司规范维度加入行业合规子项
- **引用溯源** — 每次 AI 任务记录引用了哪些行业条款

**更长期愿景**

- 多 IM 平台（企业微信 / Slack）
- 多租户 / 跨产线隔离
- 成本与可观测看板
- Skill 社区 / 跨团队分享
- **行业知识库开放生态** — 跨公司共享行业最佳实践

> 🎯 核心目标：**让 AI 懂代码、懂规范、懂业务、会学习；规范和测试做裁判，经验让平台自我进化，Skills 让能力可组合复用，行业知识让 AI 真的懂行。**
