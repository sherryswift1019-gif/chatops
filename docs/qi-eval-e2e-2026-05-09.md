## Bug 与改动

### 已修（生效中）

#### P0-1: `shouldEscalate` 调用错参，round 1 立即 escalation
[src/pipeline/graph-builder.ts:1192](../src/pipeline/graph-builder.ts#L1192)

调用 `shouldEscalate(round, loopState.budgetUsed)`，`loopState.budgetUsed` 初始为 0，导致 `1 >= 0` 永真，**round 1 直接走 escalation 子流程**（waiter 错标 `kind=escalation, decision_set=escalation`）。

修：传 `maxRounds + loopState.budgetUsed`（语义符合 PRD：budget = 基础 maxRounds + budget_extended 累计）。

```ts
// 修后
const effectiveBudget = maxRounds + loopState.budgetUsed
const approvalKind = shouldEscalate(round, effectiveBudget) ? 'escalation' : baseApprovalKind
```

复测：`shouldEscalate` 25 unit tests / `quick-impl-approval-claim` 全 pass，req#1/2 round 1 waiter `kind=spec` 正确。

#### P0-2: `mergeStageResults` 整体覆盖，吞掉 v2 结构化字段
[src/pipeline/graph-state.ts:30](../src/pipeline/graph-state.ts#L30)

`appendStageResult` 在 round 内把 `rounds[]/skillOutput/evidence/acDiff` 写进 DB，但 `skill_with_approval` 节点 resume 后返回 `finishedResult`（仅含基础字段），langgraph reducer 用 `next[existingIdx] = r` **整体替换**，把 v2 字段全清掉。最终 stage_results[1] 只剩 `{name, type, status, output, startedAt, ...}`。

后果：
- v2-ui-structured-view 折叠面板渲染不出来（acceptanceCriteria 等空）
- v2-spec-acdiff-persisted 验证不了（acDiff 丢失）
- 整个 V2StructuredView 在主路径 req#1 上空跑

修：mergeStageResults 改成浅合并 `next[existingIdx] = { ...next[existingIdx], ...r }`，保留 existing 中 incoming 未提供的 key。

复测：req#2 stage[1] 顶层有 `artifactPath/evidence/rounds/skillOutput/...`，rounds[0] 完整保留。`graph-runner` + `quick-impl-approval-claim` 共 34 unit tests 全 pass。

#### P1-1: manifest fallback 没真 fallback（与 PRD §7 R12 不符）
[src/quick-impl/skill-runner.ts:540](../src/quick-impl/skill-runner.ts#L540)

PRD R12 写"manifest 缺失/格式错 → 降级到一股脑（全 8 个 standards）+ warning log"，但实现里 `if (manifest && opts.role)` 在 manifest=null 时直接跳过，sources 集合保持空，**不创建 standards 目录**。

修：null manifest 时遍历 `docs/standards/*.md` 全部加进 sources；并在 `loadRoleManifest` 文件不存在分支补 warn log。

复测：rename manifest 后跑 req#5，worktree `.qi-context/standards/` 列出 8 个 symlink；`quick-impl-skill-runner` 28 unit tests 全 pass。

## 测试期间发现的 v2 PRD 文档与实现差异

1. **manifest fallback 行为**：PRD R12 写"全 symlink + 全字段 fallback"，实现修复前是"完全不 inject"；本次修后符合 PRD。
2. **rounds[N-1].rejectReason**：spec test SQL `stage_results->1->'rounds'->0->>'rejectReason'` 期望含 reject 文本；实现 `appendStageResult` 调用时 `decision: 'pass'`（生成器决定）不带 rejectReason，rejected 走 `loopState.rejectHistory` 而不是回写 rounds[i]。两份语义需要在 PRD/实现中对齐。
3. **spec-author selfCheck "目录为空"**：当 manifest precision injection 决定不注入某 standard 时（如 healthcheck 不需要 frontend-enum-select），spec-author 的 selfCheck 表述会写"manifest 未链接任何规范"——技术上 standards/ 仍 symlink 了 1 个文件；建议 selfCheck prompt 写得更精确（"无相关 standard 适用"而非"目录为空"）。
