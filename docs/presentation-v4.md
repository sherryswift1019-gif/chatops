## P1 · 项目定位（v4 升级）

**一句话**：覆盖 **PRD 设计 → 代码开发 → MR 审核 → 测试 → 部署运维** 的全流程平台，让 AI 做能做的，让人做该做的，**规范和测试做裁判，经验沉淀让平台自我进化**。

- **四大阶段 + 三大裁判机制**：
  1. **设计阶段** — PRD & 架构文档 AI 协作生成，规范自动校验
  2. **开发阶段** — 平台托管 或 Claude Code 本地手动，双模式并存
  3. **评审阶段** — MR 多维度质量门（需求 / 代码 / 安全 / 规范 / 测试）
  4. **运维阶段** — 部署 / 回滚 / Bug 分析修复，IM 对话式触发
- **三大裁判机制**：
  - **规范中枢**：公司 / 产品 markdown 规范动态加载
  - **测试中枢**：MR / 编译 / 发布三道强制门禁
  - **学习中枢**（v4 新）：从历史事件中持续沉淀经验，反哺未来 AI 行为
- **质量承诺**：不过门禁的代码**永远到不了生产**，同一错误**永远不犯第二次**

## P2 · 解决什么问题

**研发全链路痛点**

- PRD 质量参差不齐，评审靠人肉通读
- 公司规范散落在文档 / 老员工脑中，新人 onboarding 难
- 代码开发依赖个人习惯，多语言 / 多租户 / 安全规范执行不一致
- 测试覆盖率无硬约束，能测的测，不能测的 skip
- 测试基础设施参差，跑不起来、脚本缺失、镜像不统一
- MR review 耗时且主观，质量标准无法机器化校验
- **经验难沉淀**：bug 改了就改了、PRD 改了就改了，下一次还会犯同类错误（v4 新增痛点）
- 运维散落在 Jenkins、GitLab、K8s、Harbor，上下文切换频繁

**ChatOps 的切入点**

- **规范显性化** + **测试强约束** + **文档入 Git** + **双通道开发** + **五维 MR 门禁** + **发布前质量门**
- **自我迭代**（v4 新）：bug → 回归用例、PRD 改动 → prompt 补丁、审核失败 → 反模式手册，平台越跑越聪明
- **运维对话化**：IM 群一句话触发流水线 + 审批超时升级兜底

## P6 · 规范审核中枢

**核心理念**：规范不是文档，是**可执行的校验器**。

**规范分层**

- **公司级规范**（全产品线共享）：多语言 / 多租户 / 认证防护 / UX 交互 / UI 配色 / 日志 / 错误码 / API 契约 / **自动化测试规范** / **项目脚本规范**
- **产品级规范**（per 产品线）：领域模型、术语词典、权限模型、命名约定

**规范来源与加载**【规划】

- 存放：GitLab 知识库仓库（`product_knowledge_repos` 表）
- 格式：markdown
- 加载：审核触发时读 GitLab 最新 markdown → 注入 Agent
- **改 markdown 即生效**，版本锚定 commit sha

**使用场景（4 大卡点）**

1. PRD / 设计文档评审
2. Epic / Story 拆分评审【规划】
3. MR 代码评审
4. MR 测试评审

**工具**：`search_knowledge`【现有】、`load_company_standards`【规划】、`check_standard_compliance`【规划】

## P7 · PRD & 设计评审闭环

**原则**：**不管用户怎么写，只审核最终 markdown**。

```
编写（PRD Agent / Claude Code / 任意编辑器）
        ↓
文档进项目 Git（docs/prd/*.md、docs/arch/*.md）
        ↓
发起 MR → AI 评审（完整性 / 规范 / 可实现性 / 已有一致性）
        ↓
人工会签（产品 / 架构 / 安全）
        ↓
不通过 → 建议 → 作者改 → 重审
        ↓
合入 → 作为开发阶段输入
```

## P8 · 开发阶段：双模式并存

### 模式 A：平台托管【愿景 / 迭代中】

```
PRD + 设计（已评审）
    ↓
Epic 拆分 → Epic Review
    ↓
Story 拆分 → Story Review
    ↓
ATDD 用例生成
    ↓
Story 实现（Claude Code + ATDD 驱动）
    ↓
Code Self-Review + 本地测试
    ↓
每 Story 一个 MR → 五维质量门
```

### 模式 B：手动开发【现有】

Claude Code 本地 → 直接 MR → 同样走五维质量门

**汇聚点**：统一规范 + 统一测试约束 + 统一审计

## P9 · 平台托管开发的关键机制

- **Epic / Story 粒度**：Epic 1-2 周，Story 0.5-2 天
- **ATDD 驱动**：先写 Given/When/Then，含非功能验收（响应时间、租户隔离、权限边界），代码必须让 ATDD 通过
- **质量门内置**：拆分 Review → ATDD 检查 → 本地测试 → 提 MR
- **MR 粒度小**：一个 MR 只做一件事，review 效率翻倍

## P10 · MR 审核质量门（五维）

```
① 需求匹配度    代码 ↔ Story 验收标准
② 代码质量      可读性 / 复杂度 / 命名 / 结构
③ 安全          注入 / 泄露 / 依赖漏洞 / 认证授权
④ 公司级规范    多语言 / 多租户 / 认证 / UX / UI
⑤ 自动化测试    覆盖率达标 + 测试通过 + ATDD 执行
```

**闭环**：全过 → merge；不过 → 自动修复 Agent → 重审 ↺ → 达上限通知人工

## P11 · 自动化测试：质量基石

**覆盖率硬约束**

- 总体 ≥ **70%**，核心模块 ≥ **90%**

**基础设施规范（公司级强制）**

```

test-setup.sh           # 环境准备（幂等）
coverage.config         # 覆盖率配置
```

**三道测试门**

- **编译门**：build.sh 内嵌测试
- **MR 门**：审核 Pipeline 跑测试 + 覆盖率
- **发布门**：全量部署 + E2E + 回归

**工具**【规划】：`run_project_tests`、`check_test_coverage`、`validate_test_infra`

## P14 · AI 自我修复与自我迭代（v4 新增 · 核心）

**核心理念**：平台不是写死的规则机器，是**有记忆、会反思、能自愈**的学习型系统。每一次 bug、每一次评审失败、每一次人工 override，都是平台下一次变聪明的训练信号。

### 知识沉淀型（历史经验 → 未来 prompt）

| # | 能力 | 数据源 | 产物 |
|---|------|--------|------|
| **A1** | Bug 修复强制附 **E2E 回归用例** | bug_analysis_reports + 修复 MR | 回归测试库（防复发），发布前质量门自动跑 |
| **A2** | **PRD commit 记录 → 项目级优化提示词** | prd_documents 的 Git 修订历史 | `project_prompt_supplements`，拼进该产品线所有 AI 任务的 system prompt |
| **A3** | MR 评审失败聚类 → **代码反模式手册** | `mr_quality_reviews` 累积记录 | 注入 Review Agent 的项目专属 prompt，避免重复指出同类问题 |
| **A4** | 规范审核不通过 → **规范库自生长** | `standards_violations` 记录 | AI 自动向 GitLab 知识库提 MR 建议新增规范（人审后合入） |
| **A5** | 自动修复失败案例 → **Agent 盲区手册** | `auto_fix_history` 连续失败 N 次 | `ai_blind_spots` 表，下次遇到类似模式先报警请人工 |

### 运行时自愈 & 协作反馈

| # | 能力 | 触发 | 行为 |
|---|------|------|------|
| **B3** | 部署失败 → **自动回滚 + 根因 Bug Issue** | deploy 探针失败 | rollback + 开 bug issue（带异常栈 + 变更 diff） → 接 fix 闭环 |
| **B5** | Claude 调用异常 → **模型降级** | token 限额 / 超时 | 自动切 Sonnet / Haiku，任务完成后回升 Opus，避免卡死 |
| **D1** | 人工 override 记录 → **AI 误判样本库** | 人工修改 AI 的判断（分级 / 审核意见 / 审批） | `ai_override_samples`（反面训练信号），供后续 prompt 优化 |
| **D4** | 探索性测试反馈 → **E2E 用例孵化** | 测试人员发现新问题 | 引导同步补一条 E2E 用例进回归库 + 对应规范条目 |

### 闭环循环

```
事件发生（bug / 评审失败 / 部署失败 / 人工 override）
        ↓
    结构化入库
        ↓
  周期性 + 触发式汇总
        ↓
产物：回归测试库 · prompt 补丁 · 盲区手册 · 规范 MR · override 样本
        ↓
  注入下一次 AI 任务上下文
        ↓
     行为持续改善 ↺
```

### 为什么重要

- **价值自增长**：平台跑得越久，行为越精准，ROI 持续上升
- **不重复踩坑**：同一错误绝不犯第二次（A1/A2/A3）
- **人机协同**：人工介入的每次决策都是训练信号（D1/D4）
- **优雅降级**：异常不再是阻塞点，而是触发自愈的信号（B3/B5）

## P21 · Bug 分析 & 自动修复

## P25 · 关键工程规范

**ChatOps 平台自身**

- `resolveGitlabConfig()` 统一入口、Tool 自注册、迁移幂等、命名对齐、前端表单 Select 化

**产品项目必须遵守的公司级规范**

| 类别 | 强制要求 |
|------|----------|
| 文档 | PRD / 设计文档 markdown 进项目 git，走 MR 评审 |
| 规范载体 | 公司 / 产品规范 markdown 存 GitLab 知识库 |
| 测试入口 | `./scripts/test.sh` 统一入口，不分语言 |
| 测试覆盖率 | 总体 ≥ 70%，核心 ≥ 90% |
| 编译脚本 | **`./scripts/build.sh` 必须跑测试，测试失败 = 编译失败** |
| 打包脚本 | `./scripts/package.sh` 输出标准制品 |
| 基础镜像 | `Dockerfile.base` / `Dockerfile.test` 必备 |
| 环境准备 | `test-setup.sh` 可幂等执行 |
| **Bug 修复（v4）** | **修复 MR 必须带 E2E 回归用例** |

## P26 · 价值、成果与规划

**已实现价值**

- 统一运维入口（IM 对话式）
- AI 辅助设计（PRD / 架构）
- Bug 自动化闭环（L1-L3）
- 企业级兜底（审批 / 超时升级 / 审计）

**v2 目标**（规范 + 设计 + 开发闭环）

- 规范中枢上线
- PRD / 设计 MR 卡点
- 平台托管开发 MVP
- MR 五维质量门（含自动修复）

**v3 目标**（测试质量基石）

- 测试基础设施规范化
- MR 测试门禁 + 覆盖率硬约束
- 发布前质量门（部署 + E2E + 回归）
- 编译即测试

**v4 目标**（自我迭代学习中枢，本次新增）

- **Bug → E2E 回归用例**（A1）—— 同一 bug 绝不复发
- **PRD 修订 → 项目 prompt 补丁**（A2）—— AI 犯错不犯第二次
- **评审失败 → 反模式手册**（A3）—— Review Agent 持续变聪明
- **规范违反 → 规范自生长**（A4）—— 知识库越用越完善
- **修复失败 → 盲区登记**（A5）—— 平台知道自己不知道什么
- **部署失败 → 自动回滚 + Bug**（B3）
- **Claude 异常 → 模型降级**（B5）
- **人工 override → 样本库**（D1）
- **探索测试 → E2E 孵化**（D4）

**更长期愿景**

- 多 IM 平台（企业微信 / Slack）
- 多租户 / 跨产线隔离
- 更多 Agent 场景
- 成本与可观测看板

> 🎯 核心目标：**让 AI 做能做的，让人做该做的，规范和测试做裁判，经验让平台自我进化。**
