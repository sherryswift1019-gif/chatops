## P1 · 项目定位与行业对标

### 一句话定位

**覆盖 PRD → 开发 → MR → 测试 → 部署全流程，规范 + 测试做裁判、AI 自动化执行、经验让平台自我进化**。

### 不是另起炉灶，是整合一线实践

| Mewtwo 设计 | 业界成熟概念 | 标杆公司 |
|------------|------------|---------|
| 规范 markdown 进 Git、改 MD 即生效 | **Docs as Code** | GitLab Handbook、Microsoft Docs |
| MR AI 审核 + 自动修复 | **Automated Code Review + Quality Gates** | Google Critique、Meta Phabricator |
| 覆盖率 + 编译即测试 | **Shift-Left Testing + CI** | Martin Fowler · Google 《SE at Google》 |
| Bug → 回归用例强制 | **Defect Prevention Loop** | Google SRE Book |
| 学习中枢 | **Feedback Loop（DevOps Three Ways）** | Gene Kim 《The DevOps Handbook》 |
| Skills 商店 | **Internal Developer Platform** | Spotify Backstage（Gartner 列为 2024 top 趋势） |

### 权威数据与行业背书

- **DORA 《State of DevOps》**：精英团队部署频率、变更前置时间、失败率、恢复时间全面优于普通团队 **2-7 倍**
- **Gartner**：预测 2026 年 80% 大型软件组织建立平台工程团队
- **McKinsey《Developer Velocity at Work》**：研发效率 top 25% 企业营收增长是 bottom 25% 的 4-5 倍

## P2 · DORA 研发效能标准（核心对标）

### 什么是 DORA

**DORA = DevOps Research and Assessment**，Google Cloud 旗下研究团队。每年发布《State of DevOps Report》，基于全球 **3.6 万+** 份研发团队问卷数据，定义了衡量研发交付能力的 **4 大指标**，并将团队按表现分为 4 档：**Elite（精英）/ High（高效）/ Medium（中等）/ Low（低效）**。

这是目前**全球研发效能领域最权威、最可量化**的标准，让研发水平从"技术感受"变成"可度量数据"。

### 四大核心指标与分档标准

| 指标 | Elite 精英 | High 高效 | Medium 中等 | Low 低效 | 生活化类比 |
|------|-----------|---------|-----------|---------|---------|
| **部署频率** | 按需多次每天 | 每周-每月 | 每月-半年 | 半年以上 | 产线批次下线节奏 |
| **变更前置时间** | < 1 小时 | 1 天-1 周 | 1 周-1 月 | 1 月以上 | 从下单到交货时长 |
| **变更失败率** | 0-15% | 16-30% | 16-30% | 46-60% | 出厂产品不良率 |
| **故障恢复时间** | < 1 小时 | < 1 天 | 1 天-1 周 | 6 月以上 | 设备故障到复产 |

### 精英团队 vs 低效团队（Google 官方数据）

- 生产力高 **2 倍**
- 变更失败率低 **7 倍**
- 故障恢复快 **2600+ 倍**
- 组织商业表现（盈利 / 市场份额 / 用户满意度）显著更好

### Mewtwo 如何服务这 4 个指标

| DORA 指标 | Mewtwo 的贡献机制 |
|----------|----------------|
| **部署频率 ↑** | 自动化 MR 审核 + 测试门禁 → 小批量高频发布可行 |
| **变更前置时间 ↓** | AI 自动修复减少评审来回；规范中枢消除模糊讨论 |
| **变更失败率 ↓** | 多维质量门 + 覆盖率硬约束 + 编译即测试 拦截不合格代码 |
| **故障恢复时间 ↓** | IM 对话式运维 + Bug 自动分析 + 自动修复闭环 |

### 12 个月目标（对应 DORA Elite 水位）

- 部署频率：**日均 ≥ 1 次**
- 变更前置时间：**< 1 天**
- 变更失败率：**< 15%**
- 故障恢复时间：**< 1 小时**

**达到此水位即进入 DORA Elite 行列**——全球 **Top 10-20%** 的研发组织。

## P3 · 需求痛点（按研发全链路）

| 阶段 | 核心痛点 |
|------|---------|
| **设计** | PRD 质量参差、评审靠人肉通读；规范散落，新人 onboarding 难 |
| **开发** | 代码习惯不一，多语言 / 多租户 / 安全规范执行不一致 |
| **评审** | MR review 耗时主观，质量标准无法机器化 |
| **测试** | 覆盖率无硬约束、测试基础设施参差（脚本缺失、镜像不统一） |
| **运维** | 部署 / 回滚 / 日志散落多系统，上下文切换频繁 |
| **全链路** | AI 能力碎片化、经验难沉淀，同类错误反复犯 |

**一句话总结**：既缺**硬门禁**，也缺**软知识**，更缺**自我进化**。

## P4 · 设计理念 · 三大质量承诺

### 核心理念

> **让 AI 做能做的，让人做该做的，规范和测试做裁判，经验让平台自我进化。**

### 三大质量承诺

| 承诺 | 含义 | 类比制造业 |
|------|------|-----------|
| **不过门禁，到不了生产** | 规范 + 测试 + 评审三道卡点，自动拒绝不合格代码 | 多道工序质检，不合格品不下线 |
| **同一错误，不犯第二次** | 失败经验沉淀到学习中枢，反哺下一次 AI 行为 | PDCA 持续改进（戴明循环） |
| **能力复用，不造轮子** | 规范 + Skills + 知识库让能力跨产线复用 | 工艺知识库 + 模块化生产 |

### 一句话总结

**Mewtwo = 给软件研发装一条带智能品控的精益生产线**。

# 第二部分 · 50 人团队的现实

## P6 · 50 人团队的真实约束

### 规模画像

- 研发人数：**50 人** · 估算 5-10 条产品线 · 日均 MR **20-50 个**
- 平台可投入：**2-4 人**（占团队 4-8%）
- 决策链短，可以快速调整

### 为什么不能照抄大厂方案

| 对标公司 | 规模 | IDP 投入 |
|---------|------|---------|
| Google | 10 万+ 工程师 | 专职平台团队数百人 × 多年 |
| Microsoft One ES | 7 万+ 工程师 | 专职数百人 |
| Spotify Backstage | 数千工程师 | 专职数十人 |
| **我们** | **50 工程师** | **2-4 人** |

**问题**：直接对标大厂的 IDP 完整形态，会背上与团队规模严重不匹配的复杂度。

### 三个真实挑战

1. **投入产出比**：5 中枢 × N 子系统 × 9 自迭代 × 多 Agent，2-4 人吃不下
2. **数据密度不足**：学习中枢的 9 项大多需要百万级数据才有意义，50 人团队年均 5000-10000 个 MR 偏薄
3. **开发者体验风险**：管控越重，50 人熟人团队越容易绕过或反弹；**信任密度高的小团队，低摩擦 + 强文化** 常优于 **强管控**

## P7 · MVP 阶段不做什么

**先明确不做什么**，才能把有限资源投在高价值环节。

### 完整砍掉（1 年内不做）

| 能力 | 为什么砍 |
|------|---------|
| **平台托管开发**（Epic/Story/ATDD/多 Agent 链） | 50 人团队标准化场景不足，长链路失败风险高，ROI 不确定 |
| **Skills 商店完整形态**（4 层绑定 + 编排 Agent） | 当前 skill 数量不支撑 4 层绑定，简单配置即可 |
| **行业知识库 + RAG** | 如果只做 1 个行业，知识塞进 system prompt 就够，不必引入向量库 |
| **规范自生长 A4** | 数据不够，AI 提的规范 MR 质量参差 |
| **Agent 盲区手册 A5** | 数据不够，统计意义薄 |
| **评审反模式手册 A3** | 年度 MR 样本不足以聚类 |
| **审批规则自动优化 C1** | 规则数量有限，人工配一次即可 |
| **多 IM 平台扩展** | 钉钉 + 飞书已覆盖内部需求 |
| **多租户隔离** | 50 人一家公司，不是 SaaS |
| **成本可观测看板** | Anthropic 后台账单已能覆盖当前监控需求 |

### 简化形态（不做完整版）

| 能力 | 简化方案 |
|------|---------|
| **MR 五维并发 Pipeline 化审核** | → 单 Agent 多维合并（4 维串行审一次），不起 Pipeline 并发 |
| **发布前质量门完整版** | → 简化为"全量部署 + 冒烟"，E2E 和回归用例用 MVP 积累的 |
| **学习中枢 9 项** | → MVP 只留 A1 + B5 + D1 **三项**，其他观察期后再看 |
| **PRD 入 Git MR 卡点** | → PRD 保留 DB + PRD Agent，MR 卡点延后到 V2 |

### 这些决定的依据

**YAGNI 原则**（You Aren't Gonna Need It）：不为未来 2 年不会发生的事情做架构。

**50 人团队的护城河**不是做得多，而是**做对几件事**。

## P8 · 三件事做透（MVP 核心）

**目标**：6 个月内跑通一条产品线的研发闭环，用数据决定下一步。

### 件事 1 · 规范中枢（公司级）

- 公司级规范 markdown 托管 GitLab 知识库（第一版做 TOP 10 规范）
- 动态加载：审核触发时读取最新 MD，改 MD 即生效
- 基础工具：`load_company_standards` · `check_standard_compliance`
- **不做**：产品级规范（V2）· 行业知识库（V3）· 规范自生长（数据不够）

### 件事 2 · 测试硬约束 + 编译即测试

- 公司统一测试入口：`./scripts/test.sh`（不分语言）
- 覆盖率门禁：总体 ≥ 70% · 核心模块 ≥ 90%
- **编译即测试**：`./scripts/build.sh` 必须内嵌测试，测试失败 = 编译失败
- 基础镜像模板：`Dockerfile.base` / `Dockerfile.test`
- **不做**：发布前质量门完整版（用简化冒烟）· 行业合规测试

### 件事 3 · MR AI 审核 + 自动修复闭环

- 单 Agent 多维合并审核（需求 / 代码 / 安全 / 测试四维，一次性跑完）
- 公司规范维度用**静态规则 + 动态加载**（件事 1 输入）
- 不过 → 自动修复 Agent（复用 `fix-code`）→ push → 重审
- 3 次仍不过 → 群通知人工接管
- **不做**：5 维并发 Pipeline 化（过度工程）· 规范自生长

### 附带两项低成本自迭代

- **A1** Bug 修复强制附 E2E 回归用例（代码约定 + MR 模板检查）
- **B5** Claude 调用异常 → 模型降级（Opus → Sonnet → Haiku）

### 附带一项数据沉淀（为 V2 做准备）

- **D1** 人工 override 记录 → 样本库（不分析不反哺，先攒数据）

## P10 · MVP 成功标准（6 个月观察期）

### 硬指标（达不到就不扩展）

| 指标 | 目标 | 测量方式 |
|------|------|---------|
| 试点产线闭环跑通 | **1 条产线完整跑通** PRD → MR → 测试 → 合并 | 上线后 2 个月内 |
| MR 质量门自动审核覆盖率 | **≥ 70%** 的 MR 被自动审一次 | 月度统计 |
| 测试失败自愈成功率 | **≥ 50%** | 月度统计 |
| 覆盖率达标拦截率 | 不达标 MR **100% 拦截** | 实时 |
| MR 平均评审时长 | 下降 **≥ 30%** | 对比基线 |

### 软指标（同样重要）

- 开发者 NPS（每月做一次匿名问卷）—— **目标 > 0**（正面多于负面）
- 绕过门禁行为数 —— **目标 < 5% MR**（高说明管控太硬）
- 平台维护人力实际消耗 —— **目标 ≤ 3 FTE**（超过说明过重）

## P11 · 观察期后的决策树（V2 怎么选）

        ↓

        ↓
   ┌────────────────────┬─────────────────────┬──────────────────┐

### V2 可选项（按数据选择，不提前承诺）

- **如果 MVP 跑得顺且开发者欢迎**：扩展到 3+ 产线 + 加发布前门禁简化版 + Skills 商店 MVP（仅产线级绑定）
- **如果数据够 + 反模式清晰**：开 A2（PRD → prompt 补丁）+ A3（反模式手册）
- **如果场景多且标准化**：试点平台托管开发（单个功能）

### V3 视 V2 结果决定

- 完整 Skills 商店 / 行业知识库 / 多 IM 平台等，**1.5-2 年以后再谈**

# 第三部分 · 风险、规范与总结

---

## P13 · 风险管理

### 三大主要风险

**风险 1 · 过度工程**
- 迹象：平台维护人力超过 3 FTE、开发者抱怨系统慢、AI 调用成本爆炸
- 对策：每季度做一次"能力审查"，砍掉使用率 < 10% 的能力

**风险 2 · 开发者反弹**
- 迹象：NPS 下降、绕过门禁、"偷偷 merge"
- 对策：NPS 低时立即简化门禁 · 每个门禁都有"人工 override + 审计"逃生通道

**风险 3 · AI 依赖的稳定性**
- 迹象：Claude 限额 / 异常时系统卡死
- 对策：B5 模型降级 + 所有 AI 门禁都要有"降级到人工"的 fallback

### 关键决策授权

- 试点产品线选哪个？
- 公司规范维护归谁？（必须有明确 owner）
- 覆盖率不达标的历史代码如何处理？（建议设置过渡期）
- 自动修复失败的人工接管值班机制？

## P14 · 工程规范（产品项目强制要求）

| 类别 | 强制要求 |
|------|---------|
| **测试入口** | `./scripts/test.sh` 统一入口，不分语言 |
| **覆盖率** | 总体 ≥ 70%，核心 ≥ 90%（历史代码设过渡期） |
| **编译脚本** | `./scripts/build.sh` **必须跑测试，测试失败 = 编译失败** |
| **打包脚本** | `./scripts/package.sh` 输出标准制品 |
| **基础镜像** | `Dockerfile.base` / `Dockerfile.test` 必备 |
| **环境准备** | `test-setup.sh` 可幂等执行 |
| **Bug 修复** | 修复 MR 必须带 E2E 回归用例 |
| **规范载体** | 公司规范 markdown 存 GitLab 知识库 |

**违反后果**：规范审核不过 → MR 拒绝合入。

**过渡期政策**：历史代码 3 个月过渡期，期间覆盖率目标 ≥ 50%，之后升到 70%/90%。

## P15 · 成功度量（跨阶段共用口径）

**效率**
- MR 平均评审时长（下降率）
- PRD → 合并的 lead time
- 自动修复成功率

**质量**
- 生产事故率 / 回滚率
- 覆盖率达标产线数
- 规范违反拦截率

**使用**
- 月活跃产线数 / 月活跃用户
- 日均 IM 触发次数

**平台本身**
- 平台维护人力（目标 ≤ 3 FTE）
- 开发者 NPS（目标 > 0）
- 绕过门禁率（目标 < 5%）

**对齐 DORA 精英团队标准**（12 个月目标）
- 部署频率：日均 1 次以上
- 变更前置时间：< 1 天
- 变更失败率：< 15%
- 故障恢复时间：< 1 小时

## P16 · 总结

### 三句话总结

1. **方向对标行业一线**：Mewtwo 不是发明新东西，是把 Google / Spotify / Microsoft 一线实践整合到一个产品
2. **落地克制务实**：50 人团队只做 3 件事，用数据决定扩展
3. **聚焦而非堆砌**：明确不做什么 = 把有限资源投在高价值环节

### 核心价值主张

> **对标 DORA 精英团队标准（部署频率 / 前置时间 / 失败率 / 恢复时间），MVP 聚焦 ROI 最高的规范 + 测试 + MR 审核三件事，6 个月看数据决定是否继续扩展——既不盲目堆功能，也不错过平台化红利。**

### 执行原则

> **先把 3 件事做透，数据说话再谈扩展。**
